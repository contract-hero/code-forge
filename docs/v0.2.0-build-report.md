## Known issues

- ~~**`target_file` semantics in tests.json — present correctness issue.**~~ **Resolved in v0.3.x (PR #1, commit `ae7daf3`).** Schema split into `target_file` (source under test, kept) + `test_file` (test code path, NEW required field); `checkTestFileEditDuringGreen` now reads `test_file`; v0.3.x smoke section 11.5 has the regression guard.
- **`inferTargetFromTaskInput` heuristic for `required_subagents`.** The `Task` tool does not expose a target file as a first-class parameter, so the routing rule scans the dispatch's prompt/description for path-shaped tokens. Incidental file paths in prose can trigger false-blocks or miss legitimate routing. The `project_domains` hard rule does not have this problem (it ignores per-target inference). Fix planned for v0.4.x: extend the hook to read the *contract's* in-scope file list and match each glob against that set, instead of scanning the prompt.
- **Glob fallback YAML parsing is best-effort.** `parseRequiredSubagents` accepts the documented same-line shape (`- match: "..."`); standard YAML where `match:` lives on an indented line under `-` is currently skipped. The smoke fixtures stick to the supported shape; if a user hand-writes a different shape, the binding becomes a silent no-op. Document or upgrade the parser in v0.4.x.
- **macOS `/var → /private/var` symlink in path normalization (v0.3.x review item #5).** `forge-guard.mjs` `makeRepoRelative` does string-prefix comparison between `repoRoot` (which `process.cwd()` resolves through symlinks) and the supplied `filePath` (which Edit/Write tools may or may not resolve). On macOS, when the worktree path crosses a symlink (typical under `mktemp`, atypical for production worktrees), `relPath` falls back to the absolute path and the test-file lookup misses. Smoke fixtures realpath-resolve via `pwd -P` in `setup_green_phase_fixture`. Fix in v0.4.x: call `realpathSync(repoRoot)` and `realpathSync(dirname(filePath))` before comparison.
- **F4 protection is partial — workers can write via Bash.** `agents/implementer-worker.md`'s tool allowlist includes `Bash`. The worker prompt now (v0.3.x) explicitly forbids Bash file writes during green-phase candidate staging, but enforcement is prompt-discipline only. Hook coverage for `Bash` tool calls is a v0.4.x concern.
- **No PyYAML in the default environment** — `cycle-validate.sh`'s agent-config.md schema check falls back to presence-only (warns "PyYAML not installed; agent-config.md schema check is presence-only"). The lighter check still catches missing keys; deeper validation is opt-in via `pip install pyyaml`.
- **e2e-extract.sh fallback parser** — handles the documented YAML-in-fenced-block shape; non-trivial YAML (anchors, multi-line scalars) requires PyYAML.
- **Manual /forge dry-run on v0.3.x pending** — review item #6. Acceptance gate before merging PR #1 to main.
- **state.json has no `paused` field for failed decision gates** (surfaced by SEDEFI-176 acceptance run, see `docs/BUG-orchestrator-spawned-no-agent-tool.md` Amendment 2). Sessions ad-hoc-add `paused: true` / `paused_at` / `paused_reason` to handle G-Boot/G-Schema/G-Pyth/G-Vault halts; the schema doesn't formalize it; resume-after-pause UX is undocumented. The SEDEFI-176 session implemented a richer schema on the fly (`pause_history` array with structured timestamps + reason) — worth incorporating. Workaround: free-form fields (validator currently presence-only). Fix in v0.4.x: extend the `state.json` schema with `paused`/`pause_history`, update `forge-status.sh` to render them, and add an AskUserQuestion at the top of the resume path when `paused: true`.
- **`forge-implementer` (best-of-N coordinator) gets `Agent` tool stripped on spawn** (Amendment 3, surfaced by SEDEFI-176 Cycle 1 green). Same root cause as F6 — Claude Code subagent semantics strip `Agent`. The implementer-coordinator was supposed to fan out 6 workers via parallel `Agent` calls, but spawned without the tool, degraded to best-of-1 silently and flagged it in `synthesis-notes.md`. Workaround documented by the SEDEFI-176 session: main session reads `agents/implementer.md` as a procedure manual and dispatches the 6 workers directly. Fix in v0.4.x: apply the F6 pattern (skill-as-procedure) to `agents/implementer.md` — make it a procedure manual loaded by the orchestrating main session, not a dispatchable subagent. Note: forge-test's run produced full best-of-N (6 candidates), suggesting greenfield runs may not always hit this; SEDEFI-176's strict-spec session did. v0.4.x should fix it conclusively.
