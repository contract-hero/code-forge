Code Forge

Turn a lazy prompt into a shippable PoC. The /forge skill drives the run in your Claude Code session — Phase 0 (claudex) → Phase 1 (planner authors spec.md) → cycle loop. Each cycle spawns one claude -p /goal child that runs best-of-N implementer workers, configurable dimensional reviewers, and a consolidator. One hook keeps test files read-only during green so the cycle's /goal can't talk a worker into weakening the suite.

01What it does

Code Forge solves the three failure modes of multi-agent code generation — protocol drift, reviewer blind spots, and tautological tests — by making each one a structural property of the run rather than a prompt-level hope.

TDD as a phase

A dedicated test-author writes tests before any implementer touches code. cycle-tests-pass.sh red inverts exit codes so tautological tests fail the gate.

Best-of-N implementer

6 Sonnet workers dispatched in a single turn produce independent candidates; the cycle child scores them by LOC + files + complexity and applies the simplest passer.

Dimensional review

N reviewers (count, model, dimensions all picked in spec.md) cover a curated lens menu — correctness, design, simplicity, security, more. A consolidator clusters, verifies, ships review.md.

Goal-driven cycles

The /forge skill drives the run in your interactive session and spawns one claude -p /goal per cycle for the autonomous bits — each child loops until its result.json shows status: pass.

Anti-weakening hook

Test files (plus the state.json/tests.json anchors) are read-only during green. Without it /goal "tests pass" would mutate tests instead of the implementation.

Codex cross-checks

G2.a (spec ↔ plan) and G2.b (spec ↔ e2e) gates round-trip through Codex via codex-bridge as independent second-opinion judges.

02How a run flows

The /forge skill is the orchestrator. It runs in your interactive Claude Code session and drives Phase 0/1 + the cycle loop directly — no separate outer process. For each cycle in spec.md ## Cycle Plan, it spawns one claude -p /goal child; each child has its own Haiku evaluator that judges the cycle's goal condition after every turn.

Why this works

/goal is one-goal-per-session by design. Each cycle gets its own session — and its own evaluator — by being spawned as a separate claude -p process. The skill itself runs in your interactive session without setting /goal, so you can intervene at any point (the goal-driven autonomy lives at the cycle level, not the run level).

03Inside a cycle child

Each cycle is one claude -p /goal session whose condition is "result.json exists with status: pass AND review.md has 0 critical clusters". The child runs a linear sequence of dispatches + gates inline; the cycle's /goal handles re-prompting on failure until done or capped at K turns.

04Configurable dimensional review

Each cycle's reviewer fan-out is parameterized at spec-authoring time. The planner runs an interactive Phase 1.5 sub-step asking for the model + dimensions and writes the result into spec.md ## Reviewer Config: 

## Reviewer Config
model: opus            # or sonnet
dimensions:
  - correctness        # ★ default trio
  - simplicity         # ★ default trio
  - security           # ★ default trio

The length of the dimensions list IS the reviewer count — no separate count: field, no validation mismatch. Want two security reviewers? List security twice.

05The one hook

Code Forge ships exactly one PreToolUse hook: the test-immutability rule. Without it, /goal "tests pass" would happily mutate the tests instead of the implementation — the evaluator can only see what's in the transcript and cannot diff against a baseline.

What the hook blocks during green phase

  • Edit / Write to any path listed in the current cycle's tests.json[*].test_file.
  • Edit / Write to .forge/state.json or .forge/cycles/<id>/tests.json themselves — without this, a worker could rewrite either to disarm the rule with one Edit.
  • Bash file-writes that resolve to any of the above — covering >, >>, &>, >|, 2>, | tee, cp, mv, install, cp -t <dir>, GNU + BSD sed -i, perl -i, ruby -i, awk -i inplace, python -c open(…, "w"), dd of=, truncate, ln -sf, rm, and any sh -c / bash -c / eval / here-string invocation referencing those paths.

The hook is fail-closed: any unexpected error (malformed JSON, internal exception, etc.) exits 2 with a diagnostic instead of letting the tool call through.

Worker candidate-staging is also covered

Best-of-N workers stage candidates under .forge/cycles/<id>/green/candidates/worker-K/files/<repo-relative-path>. The hook peels that prefix so a write to a staged copy of a test file still trips.

06spec.md — the single artifact

Everything per-project lives in one file. The planner emits it during Phase 1; the skill and every cycle child read it.

  • # <Project Name> + ## Vision — what + why
  • ## Target Users
  • ## Acceptance Criteria — observable behaviors with AC-NNN ids
  • ## Architecture — concise, ~½ page max
  • ## Out of Scope
  • ## E2E Tests — YAML scenarios that the final cycle's tests cover
  • ## Cycle Plan — sequential YAML list, each entry has id, goal, files_affected, acceptance, goal_condition
  • ## Reviewer Config — model + dimensions list
Example ## Cycle Plan entry 
- id: C1
  goal: implement the user storage layer
  files_affected:
    - src/storage/users.ts
    - src/storage/types.ts
  acceptance: [AC-001]
  goal_condition: |
    cycles/C1/result.json exists with status: pass AND
    cycles/C1/review.md has 0 critical clusters,
    or stop after 30 turns

07Quick start

Code Forge is a Claude Code plugin. There's one entry point:

/forge "Build a CLI that counts source lines"

# Skip Phase 0 (Codex-mediated planning refinement) for trivial tasks:
/forge "Add a --json flag to my CLI" --quick

# Skip the optional G2.5 Codex gate (cost savings):
/forge "<description>" --light

# Resume a run that left a partial .forge/ behind:
/forge "<description>" --resume

The code-forge skill drives the whole run in your interactive session (Phase 0 → Phase 1 → cycle loop), spawning one claude -p /goal child per cycle for the autonomous bits.

Requirements

  • Claude Code v2.1.139+ — /goal is load-bearing.
  • codex-bridge plugin — provides the Codex MCP tools used by the planner's G2.a / G2.b / G2.5 cross-checks.
  • Hooks enabled — disableAllHooks: false in settings (the default). The one anti-cheat invariant is a PreToolUse hook; if it's disabled, /goal "tests pass" will reliably weaken tests.
  • jq + node in PATH — used by cycle-validate.sh and forge-guard.mjs.
